# Maycation — Experience Principles

These principles describe how the Maycation experience should feel. They apply to every screen, every interaction, and every decision about what to show or hide.

---

## One Screen Answers One Question

Every screen has a primary question. Everything on the screen should support answering that question. Information that does not support the primary question should be removed, combined, demoted, or moved to where it is actually needed.

| Screen | Primary question |
|--------|-----------------|
| My Trips | Which trip do I want? |
| Trip Dashboard | Which day do I need? |
| Day Detail | What do I need to do today? |
| Settings | What am I changing? |

If a screen cannot be described by a single primary question, it is trying to do too much.

---

## Every Interaction Reduces Uncertainty

When a user takes an action, they should feel more certain afterward, not less. Confirmations are not noise — they are evidence that something worked. Status is not decoration — it is trust.

A completed item should look complete. A saved trip should feel saved. A change made by one family member should be visible to all others without a refresh.

Uncertainty costs attention. Certainty returns it.

---

## Important Information Appears Before It Is Needed

The best moment to surface a piece of information is just before the user needs it — not while they're asking for it.

The departure time should be visible on the morning of travel. The reservation address should appear before the family leaves for the restaurant. A weather warning for tomorrow should appear today.

Proactive information is not intrusive when it is genuinely useful. The difference is timing and relevance.

---

## Assume Context

Users remember where they came from and what they just did. Maycation should too.

When a user navigates from a trip to a day, the app should not reintroduce the trip name. When a user returns to a screen they recently visited, the state should be where they left it. When a user selects a day, the app should not re-explain what a day is.

Narrating navigation insults the user. Trust that they navigated intentionally.

---

## Progressively Disclose Complexity

Lead with what is most important. Reveal depth only when the user reaches for it.

A planner item shows its name and status. The full details appear on tap. Advanced options appear when the user signals intent. The experience scales from overview to detail without overwhelming at either level.

Complexity is not the enemy. Premature complexity is.

---

## Minimize Typing

Every field that requires text input is a burden. Reduce the number of fields. Reduce the number of required fields. Prefer selection over composition wherever the options are knowable.

Where text input cannot be avoided, smart defaults, contextual suggestions, and address intelligence should do as much of the work as possible.

The goal is a complete, useful plan that required the minimum possible typing to create.

---

## Prefer Automation Over Configuration

When the system can reasonably infer something, it should. When a calculation can be performed automatically, it should be. When a default is almost always correct, it should be the default.

Configuration has a place — but only when the system genuinely cannot know what the user wants. Most of the time, it can make a reasonable guess. Let it.

---

## Prefer Suggestions Over Blank States

An empty screen is a missed opportunity.

When a trip has no days, suggest adding the first one. When a day has no items, suggest what kind of day it might be. When a form field is empty and context exists, suggest a value.

Suggestions respect the user's agency — they can be ignored or overridden. Blank states do not help.

---

## Time Is More Important Than Menus

On a travel day, a family is moving. They are in a car, an airport, a theme park. They do not have time to navigate menus to find what they need.

The closer a moment is to actual travel, the more Maycation should bring information forward rather than requiring the user to search for it. Timely context is worth more than a comprehensive menu.

---

## Use Location and Context Whenever Appropriate

Location, time of day, day of trip, and current weather are signals that make Maycation more useful without requiring additional input.

A travel day looks different from a planning day. Morning looks different from evening. Day 1 looks different from the last day. Maycation should use these signals to surface what matters at the right moment.

Context-awareness is not surveillance. It is relevance.

---

## Eliminate Duplicate Work

If a user has entered information once, they should not be asked for it again.

A trip destination entered during setup should not be re-entered in a navigation context. A reservation start time entered as a planner item should not need to be entered separately into a timeline. A family member invited once should not need to be invited again for subsequent trips.

Duplication is the clearest sign that the product is not connected.

---

## Trust the User

Maycation is for people who are planning real trips with real stakes. They do not need to be protected from their own decisions.

Confirmation dialogs should appear for destructive actions only. Navigation should not require excessive back-button chains. Forms should not require validation for fields that do not need it.

Trust the user's intent. If they made a mistake, let them undo it. Do not assume the mistake before it happens.

---

## Shared Should Feel Natural

When multiple family members are working in the same plan, coordination should feel effortless — not like a collaboration feature that was bolted on.

Changes made by one person should be visible to others. Responsibility can be distributed. No one person should be the required bottleneck for the plan to work.

The best sign that collaboration is working: the family doesn't notice it. They're just on the same page.
